OS-level blocking

Works in every browser and app on your machine. No extension to disable, no browser to switch.

Schedule-based, per group

Each domain group has its own independent schedule. Stack multiple block windows in the same day — mornings and evenings separately, or 9–12 and 2–6 on weekdays.

Tabs close automatically

Every minute during an active block, Chrome, Safari, Arc, and Brave have any tabs on blocked sites closed — even tabs opened mid-block via Private Relay or browser DoH that bypass DNS. No lingering temptations. macOS

3-minute heads-up

A native notification gives you time to wrap up before sites go dark. Friction-free transition. macOS

Live config, no restart

Edit the JSON config and save. Changes reload within 60 seconds — no daemon restart, no downtime.

Survives sleep & wake

Close the lid and open it hours later. The scheduler catches up on the next tick — no stale state, no gaps.

Web dashboard at :8040

Manage rules, check blocked status, test schedules, and pause blocking — all at localhost:8040.

Daily quotas (allowance mode)

Cap how long a group is accessible each day. Add daily_quota_minutes: 30 to any rule — once the allowance is consumed, the group blocks for the rest of the day. Requires DNS or strict mode.

Focus sessions (Pomodoro)

Start a 25/5 work-and-break session from the dashboard. During work, every active rule is forced on regardless of schedule and pause is locked out. Notifications fire at each phase boundary.

Clean uninstall

One command removes every system change — hosts entries, service registration, DNS config, firewall rules. Leaves nothing behind.

Sentinel is enforcing

Three levels of enforcement

Start with the default. Escalate if you need to.

Default

Hosts

Edits /etc/hosts (macOS) or the Windows hosts file. Blocked entries are atomically rewritten between Sentinel-managed markers — other entries untouched.

  • ✓ No port binding
  • ✓ Works in every app
  • ✓ macOS + Windows
  • ✗ No wildcard subdomains
sudo sentinel --set-mode hosts
Advanced

DNS

Runs a local DNS proxy on 127.0.0.1:53. Blocked domains return 0.0.0.0; everything else forwards to your upstream resolver with automatic failover.

  • ✓ Wildcard subdomain matching
  • ✓ macOS + Windows
  • ✓ Primary + backup DNS failover
  • ✗ Requires OS DNS redirect
sudo sentinel --set-mode dns
Strict

Strict

DNS proxy plus a pf (Packet Filter) anchor that drops outbound packets to blocked IPs at the kernel level. For when DNS isn't enough.

  • ✓ Kernel-level IP blocking
  • ✓ Beats hard-coded resolvers
  • ! macOS only
  • ✗ Degrades to DNS if pf fails
sudo sentinel --set-mode strict

FAQ

The honest answers.

AdGuard Home is a solid network-wide DNS blocker, and Sentinel can run alongside it. But its scheduling model has hard limits that make fine-grained personal rules impractical:

Capability Sentinel AdGuard Home
Multiple block windows per day per group ✓ e.g. 9–12 and 2–6 One slot per day
Independent schedule per domain group ✓ Per rule One shared schedule
Custom domain groups ✓ Any domains Predefined catalog only
Browser tab auto-close on block ✓ macOS ✗
Pre-block notifications ✓ macOS ✗

AdGuard Home excels at network-wide content filtering — blocking ad trackers or adult content for every device on a home network. Sentinel is built for personal schedule enforcement on a single machine: granular enough to say "block Reddit 9–12 and 2–6, block gaming all evening, and leave streaming open on weekends." The two tools serve different needs and can run alongside each other.
